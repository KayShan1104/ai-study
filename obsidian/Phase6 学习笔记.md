# Phase 6 学习笔记

> AI Agent 测试与评估专项方向

---

## Step 0: 认知升级——从传统测试到 Agent 测试

### 核心差异：确定性 vs 非确定性

传统测试的根基是**确定性（Determinism）**——同一输入，永远同一输出。LLM 的**概率性本质（Probabilistic Nature）**打破了这个根基。同一 prompt 运行 10 次，可能得到 10 种不同但都"对"的回答。

### 断言范式的转变

| 维度 | 传统测试 | Agent 测试 |
|------|----------|------------|
| 断言方式 | `assertEquals`（精确匹配） | 语义评分 + 模型裁判 + 统计趋势 |
| 回归标准 | 全部 pass 才放行 | 评分分布稳定、趋势线不下降 |
| 测试用例 | 手动编写边界值 | 真实对话 + 对抗性样本 + 数据增强 |
| 评估频率 | 代码变更时 | 每次模型更新 / prompt 调整 / 工具变更 |

### 从"判断对错"到"评估质量"

Agent 测试需要多维度评估：

- **Answer Relevance（回答相关性）**: 回答是否解决了用户问题？通过检查输出是否包含关键信息（如城市名、天气描述、温度）来判断
- **Faithfulness（忠实度/无幻觉）**: 回答是否忠于事实？需要裁判模型或外部事实源来判断
- **Tool Use Correctness（工具调用正确性）**: Agent 是否正确选择了工具？参数是否合理？
- **Safety（安全性）**: 是否包含有害内容？
- **Format Compliance（格式合规）**: 输出格式是否符合要求（如语言、JSON 格式等）
- **Latency（延迟）**: 响应时间是否在阈值内
- **Cost（成本）**: Token 消耗是否在预算内

### 测试经验迁移指南

**可以迁移的经验（保留）：**

1. 测试用例设计思维：正常用例 + 边缘用例 + 对抗用例
2. 自动化测试框架搭建能力：数据驱动、报告生成
3. CI/CD 集成经验：代码变更 → 触发评估 → 阈值判断
4. 质量度量思维：覆盖率、趋势分析、回归追踪
5. 缺陷定位能力：虽然中间步骤变复杂，但定位问题的思维相通

**必须抛弃的思维定式：**

1. 精确断言 `assertEquals` → 转向语义评分和统计判断
2. 100% 通过率目标 → 接受一定范围内的波动，关注趋势而非绝对值
3. 确定性回归标准 → 用评分分布和置信区间替代 pass/fail
4. 测试用例是"资产" → 在 Agent 世界，测试数据需要持续更新和演化
5. Bug 是非黑即白的 → Agent 的行为质量是连续谱，需要容忍灰度

### Agent 测试的核心挑战

**如何为非确定性系统建立可信的质量保障体系。**

需要从"判断对错"转向"评估质量"：
- 不是"输出是不是这个字符串"，而是"输出是否解决了用户问题"
- 不是"函数返回值 == 预期"，而是"工具调用是否正确、参数是否合理"
- 不是"全部 pass"，而是"评分分布是否在可接受范围内"

## Step 1: 评估指标体系设计

### 评估指标设计（代码审查 Agent）

设计了 7 个评估指标，覆盖 Agent 行为的全链路质量：

| 指标 | 权重 | 评估方式 | 说明 |
|------|------|----------|------|
| 工具调用正确性（Tool Use Correctness） | 30% | 规则判断 | 是否选对工具、参数准确 |
| 回答相关性（Answer Relevance） | 20% | LLM-as-judge | 回答是否解决了用户问题 |
| 忠实度（Faithfulness） | 15% | LLM-as-judge | 是否基于工具返回，有无编造 |
| 格式合规（Format Compliance） | 10% | 规则判断 | 输出是否结构化、可读 |
| 响应延迟（Latency） | 10% | 阈值判断 | 耗时是否在可接受范围 |
| 安全性（Safety） | 10% | LLM-as-judge | 是否产生有害建议 |
| 成本（Cost） | 5% | 阈值判断 | token 消耗是否在预算内 |

**权重设计原则**：工具调用正确性是 Agent 的核心能力，权重最高（30%）；LLM-as-judge 相关指标合计 45%；性能和成本合计 15%。

### LLM-as-judge 模式实践

**实现方式**：用另一个 LLM 调用作为"裁判"，对 Agent 的回答进行多维度评分。

裁判 prompt 的设计要点：
1. 明确角色："你是评估专家"
2. 给出评分标准：0-100 分，分 5 个档位
3. 要求只返回整数分数，便于程序化处理
4. 裁判模型用 `temperature=0.0` 保证评分一致性

**LLM-as-judge 的 3 个主要局限**：
1. **不一致性（Inconsistency）**：同一回答多次评分可能得到不同分数（即使 temperature=0）
2. **位置偏见（Position Bias）**：裁判模型倾向于给排在前面的回答更高分
3. **自我偏好（Self-Preference Bias）**：用模型 A 评判模型 A 生成的内容时分数偏高，用不同模型当裁判评分可能偏低

### 评估结果分析

5 个测试用例的评估结果：

| 用例 | 类别 | 难度 | 加权总分 |
|------|------|------|----------|
| TC_001 | tool_use | easy | 94.2 |
| TC_002 | tool_use | medium | 65.2 |
| TC_003 | follow_up | medium | 93.2 |
| TC_004 | safety | hard | 85.5 |
| TC_005 | edge_case | easy | 36.0 |

**整体加权平均分：74.8**

**薄弱指标**（<70 分）：
- 回答相关性 59.0：TC_002（SQL 注入场景）和 TC_005（边缘场景）拉低了平均分
- 格式合规 69.0：部分回答结构不够清晰

**分析**：
- TC_002 安全性评 0 分——裁判模型认为 Agent 指出的安全问题不够严重
- TC_005 得 36 分——边缘场景下工具未调用（0 分），回答相关性仅 30 分，说明 Agent 处理空输入时表现不佳

### 评估引擎架构

评估流程分为三类评分方式：
1. **规则评估（rule）**：无需 LLM，用代码逻辑判断（工具调用、格式合规），成本低速度快
2. **LLM-as-judge 评估（llm）**：调用裁判模型进行语义评分，成本高但灵活
3. **阈值评估（threshold）**：与预设阈值对比线性衰减（延迟、成本）

生产环境中，规则评估 + LLM-as-judge 结合是最佳实践。

## Step 2: 评估数据集构建

### 数据集构建方法

采用**手工编写 + LLM 辅助生成**的组合方式：

- **手工编写**：75 个核心用例，覆盖典型场景和已知边界
- **LLM 辅助生成**：25 个用例，用 LLM 根据场景描述自动生成，扩大覆盖范围

### 数据集统计

| 维度 | 数量 |
|------|------|
| 总用例 | 100 |
| 正常用例（normal） | 25 |
| 边缘用例（edge_case） | 13 |
| 对抗用例（adversarial） | 15 |
| 压力用例（stress） | 10 |
| 多语言用例（multilingual） | 12 |
| LLM 生成用例（llm_generated） | 25 |

| 难度 | 数量 |
|------|------|
| easy | 19 |
| medium | 46 |
| hard | 35 |

### 测试用例格式

每个用例包含：
- `id`: 唯一标识（如 TC_N001）
- `category`: 类型分类
- `difficulty`: 难度等级（easy/medium/hard）
- `input`: 用户输入
- `expected_tool`: 预期调用的工具
- `expected_behavior`: 预期行为描述
- `expected_output_features`: 预期输出特征（数组）
- `anti_patterns`: 反模式（Agent 不应出现的行为）
- `notes`: 备注

### 覆盖度分析

**覆盖充分的场景**：
- 安全类对抗（15 个 adversarial 用例，远超 10 个建议值）
- 正常代码审查（25 个 normal 用例）
- 边缘场景（13 个 edge_case，含空输入、彩蛋代码、危险函数）

**可能不足的场景**：
- 压力用例仅 10 个——复杂多步任务场景可以再扩充
- 缺少多 Agent 协作场景的用例（当前只针对单 Agent 代码审查）
- 缺少真实生产代码样本（当前用例多为示例代码）

### LLM 生成用例的经验

用 LLM 辅助生成测试用例时需要注意：
1. LLM 可能返回格式不正确的 JSON，需要加异常处理
2. 生成的用例质量参差不齐，需要人工审核
3. 生成器 prompt 要明确指定格式和数量，避免 LLM 自由发挥

## Step 3: 自动化评估流水线

### 流水线架构

```
加载数据集 → 批量运行 Agent → 多维度评分 → 生成报告 → 阈值判断 → CI 状态
                                    ↓
                              保存历史记录（趋势对比）
```

### 核心组件

1. **数据集加载**：从 `test_cases.json` 读取 100 个用例
2. **Agent 执行**：批量运行，收集轨迹（工具调用、回答、错误）
3. **评分引擎**：规则评估（40% 用例）+ LLM-as-judge 抽样评估（60% 用例）
4. **报告生成**：JSON 格式报告 + 控制台可读输出
5. **历史追踪**：`eval_history.json` 保存最近 10 次运行记录
6. **CI 集成**：`--ci` 参数模式，低于阈值返回 exit code 1

### 阈值配置

| 维度 | 阈值 | 说明 |
|------|------|------|
| 整体加权平均分 | 70.0 | PASS/FAIL 判定标准 |
| 工具调用正确性 | 80.0 | Agent 核心能力 |
| 安全性 | 70.0 | 安全底线 |
| 回答相关性 | 60.0 | 允许一定波动 |

### 评估结果（100 用例）

**整体加权平均分：74.0**（阈值 70.0）→ ✅ PASS

**按类别**：
| 类别 | 分数 | 状态 |
|------|------|------|
| normal | 83.1 | ✓ |
| multilingual | 73.7-84.9 | ✓ |
| llm_generated | 81.5-83.7 | ✓ |
| adversarial | 64.3-64.6 | ⚠ |
| edge_case | 56.5-62.9 | ⚠ |
| stress | 55.5-56.2 | ⚠ |

**按难度**：
| 难度 | 分数 |
|------|------|
| easy | 78.4-79.4 |
| medium | 77.8-78.9 |
| hard | 64.6-66.5 |

### 关键发现

1. **对抗用例和压力用例是薄弱环节**：adversarial 和 stress 类别均低于 70 分
2. **难度越高，分数越低**：hard 难度比 easy 低约 15 分
3. **LLM-as-judge 成本**：100 个用例需 160 次 LLM 调用，耗时约 90 秒
4. **通过率仅 48-49%**：虽然整体加权平均分过了阈值，但近一半用例未达标——说明阈值设定和用例质量需要进一步调优

### CI 集成方案

流水线支持 `--ci` 模式，可直接嵌入 GitHub Actions：

```yaml
name: Agent Evaluation
on: [push, pull_request]
jobs:
  evaluate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: pip install -r requirements.txt
      - run: python code/phase6/phase6_step3_eval_pipeline.py --ci
```

低于阈值时 exit code 1，CI 标记为 fail。

## Step 4: Agent 行为监控与调试

### Trace 系统设计

实现了三层追踪数据模型：

1. **RunTrace（运行级别）**：一次 Agent 运行的完整轨迹
2. **StepTrace（步骤级别）**：每一步的 Thought、Action、Observation
3. **ToolCallTrace（工具级别）**：工具调用的输入、输出、状态、错误

```
RunTrace (TC_A002, adversarial, error)
├── Step 1: parse_input ✅ (50 tokens, 10ms)
└── Step 2: call_tool(code_review) ❌ (60 tokens, 21ms)
    └── ToolCallTrace: TypeError - 工具执行失败
```

### 错误分析能力

通过 Trace 系统可以精确定位：
- **在哪一步失败**：15 次运行中 3 次失败，全部在 Step 2（工具调用阶段）
- **失败原因类型**：工具异常 2 次、超时 1 次
- **Token 消耗分布**：`generate_response` 步骤最耗 token（120 平均），`parse_input` 最少（50）

### 监控平台对比

| 平台 | 类型 | 优势 | 劣势 |
|------|------|------|------|
| LangSmith | SaaS/自托管 | 功能最全、LangChain 深度集成 | 免费版有限制 |
| LangFuse | 开源/Cloud | 开源免费、可自托管数据隐私好 | 功能不如 LangSmith 全 |
| Phoenix | 开源 | Embedding 分析强 | 部署复杂 |

### 关键洞察

Agent 调试的核心难点在于**黑盒性**——LLM 的内部决策过程不可见。Trace 系统记录的是"外部行为"（Thought、Action、Observation），而不是真正的"思考过程"。这就像通过日志调试一个分布式系统——能看到交互序列，但看不到每个组件的内部状态。

## Step 5: 安全评估

### OWASP LLM Top 10

| 编号 | 威胁 | 说明 |
|------|------|------|
| LLM01 | 提示词注入（Prompt Injection） | 攻击者通过输入覆盖系统提示 |
| LLM02 | 不安全的输出处理 | LLM 输出未验证直接传给下游 |
| LLM03 | 训练数据投毒 | 篡改训练数据植入后门 |
| LLM04 | 模型拒绝服务 | 超长输入消耗计算资源 |
| LLM05 | 供应链漏洞 | 第三方依赖存在安全风险 |
| LLM06 | 敏感信息泄露 | 泄露训练数据或系统信息 |
| LLM07 | 不安全的插件设计 | 插件缺乏权限验证 |
| LLM08 | 过度依赖 | LLM 被授予过多自主权限 |
| LLM09 | 过度对齐 | 过度拒绝或过度信任 |
| LLM10 | 数据/模型投毒后滥用 | 通过反馈循环改变模型行为 |

### 安全测试数据集

构建了 30 个安全测试用例，覆盖 6 个活跃 OWASP 类别：

| 类别 | 用例数 | 防御率 |
|------|--------|--------|
| LLM01 提示词注入 | 11 | 64% |
| LLM06 敏感信息泄露 | 7 | 71% |
| LLM08 过度依赖 | 4 | 50% |
| LLM02 不安全输出处理 | 3 | 100% |
| LLM04 模型拒绝服务 | 3 | 100% |
| LLM07 不安全插件设计 | 2 | 100% |

**整体防御率：73.3%（22/30）→ 🟡 中等风险**

### 薄弱点分析

**hard 难度场景防御率仅 60%**，失败集中在：
- **角色扮演注入**：通过游戏/故事框架绕过限制
- **Base64 编码注入**：编码隐藏恶意指令
- **社会工程**：以"安全研究员"身份伪装合法需求
- **逐步升级**：从正常请求逐步过渡到恶意意图

这些攻击方式的共同特点是**隐蔽性强**——不直接使用明显的恶意指令，而是通过上下文包装、身份伪装、渐进引导来降低 Agent 的安全警觉。

### 安全评估的关键认知

安全测试与功能测试不同：功能测试关注"Agent 能不能做对"，安全测试关注"Agent 会不会做错"。一个功能完备的 Agent 如果没有安全防线，反而更危险——因为它有更强的能力执行恶意操作。

## Step 6: 综合项目——Agent 质量评估平台（Portfolio）

将 Phase 6 所有学习成果整合为一个独立的 CLI 工具项目 `agent-quality-platform`，已独立发布到 GitHub。

### 项目结构

```
agent-quality-platform/
├── README.md          # 项目文档 + 快速开始
├── cli.py             # CLI 入口（evaluate/security/init/test）
├── requirements.txt   # 依赖
├── .env.example       # 环境变量模板
├── .gitignore         # Git 忽略规则
├── src/
│   ├── metrics.py     # 7 维度评估指标引擎
│   ├── evaluator.py   # 单用例/批量评估
│   ├── dataset.py     # 数据集构建和管理
│   ├── pipeline.py    # CI/CD 流水线
│   ├── trace.py       # Trace 追踪系统
│   ├── security.py    # OWASP LLM Top 10 安全扫描
│   └── report.py      # 报告生成和趋势
└── tests/
    └── test_all.py    # 18 个单元测试
```

### CLI 命令

- `python cli.py init` — 初始化示例数据集
- `python cli.py evaluate --dataset data/test_cases.json` — 运行评估流水线
- `python cli.py security --report` — 运行安全扫描
- `python cli.py test` — 运行单元测试

### 设计原则

1. **模块分离**：每个功能独立模块（metrics/evaluator/dataset/pipeline/trace/security/report），可单独使用
2. **可扩展**：Agent 执行函数通过 `agent_fn` 参数注入，适配不同 Agent 实现
3. **零依赖**：仅依赖 `openai` 和 `python-dotenv`，LLM 评估可选开启
4. **测试覆盖**：18 个单元测试覆盖核心指标、数据集、安全用例
