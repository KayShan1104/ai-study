## Step 1：理解 Embedding（嵌入）

### Embedding 是什么

Embedding（嵌入）是将一段文本转换为一组浮点数（向量），这组数字捕捉了文本的**语义信息**。语义相近的文本，它们的向量在向量空间中的距离就近。

本实验使用 `text-embedding-v3` 模型，输出维度为 **1024**——每段文本被编码为 1024 个浮点数，共同捕捉语义信息。

### 实验数据

10 段文本分为 4 组（编程、天气、AI、食物），计算组内和跨组的余弦相似度：

| 文本对 | 组别 | 相似度 |
|--------|------|--------|
| "我爱编程" vs "我喜欢写代码" | 编程（同组） | **0.8737** |
| "我爱编程" vs "Python 是一门好语言" | 编程（同组） | 0.6592 |
| "今天天气很好" vs "外面阳光灿烂" | 天气（同组） | **0.7734** |
| "深度学习" vs "神经网络" | AI（同组） | 0.6102 |
| "这道菜很好吃" vs "火锅是我的最爱" | 食物（同组） | 0.6214 |
| "今天天气很好" vs "这道菜很好吃" | 跨组 | 0.6758 |
| "我爱编程" vs "深度学习" | 跨组 | 0.4575 |
| "我喜欢写代码" vs "这道菜很好吃" | 跨组 | 0.4938 |

### 余弦相似度（Cosine Similarity）

把 embedding 看作高维空间中的一个方向（向量），余弦相似度 = 两个向量夹角的余弦值。

- 夹角 0° → 相似度 1.0（完全一致）
- 夹角 90° → 相似度 0.0（无关）
- 范围 [-1, 1]，越接近 1 表示语义越相似

Embedding 不是关键词匹配，而是理解"意思"——"我爱编程"和"我喜欢写代码"没有共同关键词，但语义相似度最高（0.87）。

### 短文本基线相似度偏高的原因

实验中发现一个反直觉现象："今天天气很好" vs "这道菜很好吃" 相似度 0.68，比同组内某些 pair 还高。虽然两句话语义完全不相关。原因有三：

**1. 句法结构相似**：Embedding 模型不仅编码语义，也编码句法结构。两句话的模式几乎一样（`[主语] + [副词] + [形容词]`），在向量空间中会被拉到一起。

**2. 情感极性相同**：两句都是正面评价。Embedding 模型在训练过程中吸收了情感信息——正面情感的句子在某些维度上趋向一致。

**3. 高频共用词的主导效应**：中文短句子（4-7 字）大量共用"的"、"是"、"很"等高频词。这些词在向量空间中占据主导地位，导致不同主题的短句也会有一个"基线相似度"。

编程 vs 天气（0.54）、编程 vs 食物（0.49）——这个 0.5 左右的"底"就是短句共有的结构/情感/高频词带来的。真正有意义的判断不是"0.68 高不高"，而是**相对差距**——同组 0.87 vs 跨组 0.68，差 0.19，说明模型确实区分了语义，只是短句区分度有限。

### Q&A：跨组 0.46 说明什么

**Q**: "我爱编程" vs "深度学习在图像识别中很厉害" 相似度 0.46，这说明模型认为这两句话是"完全不相关"还是"有一定关联"？

**A**: 模型认为**有一定关联**，但不算接近。0.46 处于中间偏下的位置——远低于同组内的 0.87，但高于随机句子之间的预期相似度。这是因为：
- 两者都属于"技术/编程"大领域，模型见过大量同时出现这些话题的文本，学到了一些共享特征
- 但主题差异明显：一个是"编程爱好"，一个是"AI 应用"，在更细粒度的语义维度上方向不同
- 在 RAG 检索场景中，0.46 通常不会被召回（top-k 阈值一般更高），说明模型做了有效区分

### Q&A：为什么语义相近的向量在空间中距离近？

**Q**: 为什么语义相近的向量在空间中距离近？是天生如此还是训练出来的？

**A**: **是训练出来的，不是天生的。**核心机制在于 embedding 模型的训练目标——对比学习（Contrastive Learning）：
- 训练数据包含正样本对（语义相似的句子，如"我爱编程"和"我喜欢写代码"）和负样本对（不相似的句子）
- 模型把句子编码成向量后，计算正样本对的距离——如果距离远，产生大 loss
- 计算负样本对的距离——如果距离近，产生大 loss
- 反向传播更新参数，让正样本对靠拢、负样本对推开
- 重复几十亿次，模型学会把语义相近的句子映射到空间中相近的位置

可以类比**磁极**：训练数据告诉模型"哪些句子应该互相吸引，哪些应该排斥"。1024 个维度中的每一个，都对应着模型从训练数据中学到的某种语义特征（比如"是否与编程相关"、"是否正面情感"）。

训练前向量是随机分布的，没有规律；训练后语义相近的句子被"拉"到一起，形成语义簇。

### Q&A：如果训练数据中"编程"和"AI"经常同时出现会怎样？

**Q**: 如果训练数据中"编程"和"AI"经常同时出现，模型会把它们拉近还是保持距离？

**A**: **会拉近，但不会重叠。**"编程"和"AI"会被拉到同一语义簇（技术领域）附近，但各自保持独立位置。类比两个相邻的街区：都在"科技"这个大区，但"编程"街和"AI"街还是有各自的位置。实验中 0.46 的相似度正好印证了这一点——模型知道它们是邻居，但不是同一条街。

### Q&A：Embedding 的维度是固定的吗？1024 维度代表什么？

**Q**: embedding 有 1024 个维度，这个是固定的吗？为何是 1024？这些维度代表了什么？

**A**: 
- **1024 不是固定的**，是模型设计时决定的。不同模型维度不同：OpenAI text-embedding-3-small 是 1536 维，BERT base 是 768 维，all-MiniLM-L6-v2 是 384 维。阿里云 text-embedding-v3 还支持通过参数指定输出维度（如 512、256），做降维投影。
- **单个维度没有人类可读含义**。不是"第 42 维 = 是否与编程相关"，而是 1024 个维度协同工作，共同定义向量在 1024 维空间中的位置。类比照片像素：单个像素看不出什么，组合起来才能表达完整画面。
- **1024 是效果与成本的权衡结果**。维度越大表达能力越强，但存储/计算成本也越高。1024 还是 2 的幂（2^10），在 GPU/TPU 硬件上计算效率最高。

### Q&A：维度越多检索效果一定越好吗？

**Q**: 如果有个 10000 维的 embedding 模型，检索效果一定比 1024 维的更好吗？

**A**: **不一定，甚至可能变差。** 原因：
- **维度灾难（Curse of Dimensionality）**：在高维空间中，所有向量之间的距离趋于相似，区分度反而下降。10000 维时，最相似和最不相似的两个向量，距离差距可能还不如 1024 维时明显。
- **噪声维度**：如果模型的参数量和训练数据没有跟着增长，硬拉到 10000 维，多出来的维度学到的不是语义，而是训练数据中的随机噪声。结果向量更稀疏，更容易匹配到不相关内容。
- **实际工程做法**：维度不是越多越好，而是匹配模型能力和任务需求。OpenAI 把 text-embedding-3-large 从 3072 维默认降到 1024（可指定更高），因为实验发现 1024 在效果和成本之间是最优解。

### Q&A：高维 embedding 中各维度有权重吗？

**Q**: 当维度变高时，各个维度会有权重吗？维度不是人类可读的，但直觉上应该有个权重因素。

**A**: **直觉是对的，有"隐式权重"**。虽然 embedding 向量本身没有显式的权重参数，但每个维度的值大小就是它的"信号强度"：
- 对当前句子语义敏感的维度，值会比较大（如 0.82）→ 在余弦相似度计算中自然贡献大
- 对当前句子不敏感的维度，值接近 0（如 0.001）→ 几乎不贡献，相当于权重为 0
- 余弦相似度 = Σ(ai × bi)，每个维度数学上平等参与，但因为 ai 值差异巨大，实际贡献不均
- 这就像模型训练时学会了对不同维度"分配信号强度"，向量值本身扮演了权重角色

这也解释了为什么维度太多反而不好——信号分散到 10000 个维度上，每个维度值都变小，信噪比降低。集中在 1024 个维度上，信号更强，区分度更好。

- **不是关键词匹配**：没有共同字也能找到语义相近的内容
- **向量空间中的距离 = 语义距离**：同领域、同主题、同情感的句子在空间中聚成簇
- **短句区分度有限**：句子越短，共用结构/高频词的影响越大，相似度基线越高
- **相对差距比绝对值更重要**：判断"语义是否接近"要看同组 vs 跨组的差距，而不是单看一个数值

## Step 2：Vector Database 基础

### Vector DB 是什么

Vector DB（向量数据库）专门用于存储和检索海量向量数据。它的核心价值：

- **存储**：保存文档的 embedding 向量及对应的原始文本/元数据
- **检索**：用索引结构实现高效的相似性搜索，避免 O(n) 的暴力遍历
- **过滤**：支持 metadata filter（按标签、日期等条件过滤后再检索）

本实验使用 **Chroma**，一个轻量级 Python 向量数据库，适合本地开发。生产环境常用 Pinecone（托管）、Qdrant（Rust）、Milvus（大规模）、Weaviate。

### Chroma 的工作方式

```python
# PersistentClient 持久化到磁盘，Client 是纯内存的
chroma_client = chromadb.PersistentClient(path="./chroma_data")
collection = chroma_client.create_collection(name="tech_docs")
collection.add(documents=texts, embeddings=embeddings, ids=ids)
```

- 需要手动生成 embedding 后传入，或用自定义 embedding 函数
- 支持删除集合、更新文档等操作
- 查询时返回 top-k 结果及距离值

### HNSW 索引（Hierarchical Navigable Small World）

Vector DB 为什么能"快"——核心在于 HNSW 索引：

- **多层跳表结构**：类似跳表（Skip List），上层节点稀疏，下层稠密
- **搜索过程**：从顶层随机节点开始，逐层向下找"最近的邻居"
- **复杂度**：从 O(n) 暴力遍历降到 O(log n)，百万级向量也能毫秒级返回
- **近似搜索**：HNSW 是近似算法，牺牲极小精度换取巨大速度提升
- **参数**：`M`（每节点最大连接数）和 `ef_construction`（构建时搜索宽度）控制速度和精度的权衡

可以类比**城市道路导航**：先上高速（顶层，跨越大区域），再走国道（中层），最后进小区道路（底层，精确到目标）。

> **关联**: [[Phase3 学习笔记#余弦相似度（Cosine Similarity）]] — HNSW 搜索时用的距离度量可以是余弦相似度、L2 距离、内积等，Chroma 默认用 L2 distance（距离越小越相似）。

### 语义搜索 vs 关键词搜索

实验对比了两种搜索方式对同一查询 "Python 的 Web 框架有哪些" 的结果：

**关键词搜索**（匹配包含 "Web"、"框架"、"Flask"、"Django"、"FastAPI" 的文档）：
- 返回 8 篇文档，其中 5 篇与 Web 框架无关（如 Git、Docker、Nginx 等包含关键词）
- 问题：字面匹配，不理解语义，召回了大量不相关内容

**语义搜索**（embedding 向量距离）：
- Top 3：Django（距离=0.2846）、Flask（0.3102）、FastAPI（0.3518）
- 准确命中了 3 个 Web 框架，没有无关结果
- 文档中即使没有"Web 框架"四个字，只要描述的是这个概念，就能被召回

**为什么"Python Web 框架"能搜到 Flask/Django/FastAPI**：
- Embedding 模型在训练时见过大量"Flask"和"Web 框架"同时出现的文本
- 对比学习（[[Phase3 学习笔记#Q&A：为什么语义相近的向量在空间中距离近？|Contrastive Learning]]）让它们在语义向量空间中距离接近
- 查询向量和文档向量在高维空间中距离近 → 被检索到

### 语义搜索实验结果（22 篇技术文档，5 个查询）

| 查询 | Top 3 结果 | 分析 |
|------|-----------|------|
| "Python 的 Web 框架有哪些" | Django, Flask, FastAPI | 准确命中 3 个 Web 框架 |
| "如何部署 Python 应用" | Gunicorn, Flask, Nginx | Gunicorn 是 Python WSGI 服务器，最相关；Flask 是部署对象；Nginx 常用于反向代理 |
| "怎么保证 API 安全认证" | JWT, OAuth 2.0, RESTful API | JWT 和 OAuth 是认证方案，RESTful API 是设计规范（相关性较弱但语义接近） |
| "异步编程怎么做" | asyncio, FastAPI, Celery | asyncio 是标准库，FastAPI 支持异步，Celery 是异步任务队列 |
| "怎么自动测试和发布代码" | pytest, GitHub Actions, Git | 测试框架、CI/CD 工具、版本控制 |

### 关键词搜索 vs 语义搜索的核心差异

| 维度 | 关键词搜索 | 语义搜索 |
|------|-----------|---------|
| 匹配方式 | 字面匹配 | 向量距离 |
| 新词/同义词 | 无法处理 | 自动理解同义表达 |
| 召回精度 | 容易召回无关结果 | 按语义相关排序 |
| 计算复杂度 | O(n) 字符串比较 | O(log n) 向量索引 |
| 依赖 | 无 | 需要 embedding 模型 |

### RAG 检索优化方案

**语义漂移（Semantic Drift）**：查询核心意图是"安全认证"，但被"API"这个词带到了设计规范方向（RESTful API 被错误召回）。生产中有 4 种解法：

**1. Query Rewrite / Expansion（查询重写/扩展）**

让 LLM 把模糊 query 拆成具体子问题，分别检索后合并：
```python
# 原始: "怎么保证 API 安全认证"
# 重写: ["JWT token 验证", "OAuth 2.0 授权流程", "API key 管理"]
```
优点：召回精准；缺点：多一次 LLM 调用，增加延迟和成本。

**2. Hybrid Search（混合检索）**

原始 query 同时走 BM25 关键词 + 向量语义搜索，然后算法合并：
```python
keyword_results = bm25.search("怎么保证 API 安全认证")  # 精确匹配
semantic_results = vector.search("怎么保证 API 安全认证")  # 语义理解
final = rrf_merge(keyword_results, semantic_results)
```
- BM25 自动分词并根据 IDF（逆文档频率，Inverse Document Frequency）加权，稀有词贡献更大，**不需要单独提取关键词**
- 合并算法常用 **RRF（Reciprocal Rank Fusion，倒数排名融合）**：`score = 1/(rank + 60)`，在两个列表中排名都靠前的文档得分最高
- 纯数学合并，**不涉及 LLM**，零额外成本
- 优点：互补盲区；缺点：权重参数需要调

**3. Reranking（重排序）**

先放宽召回（top 20），再用专门的 Cross-Encoder reranker 模型精排 top 3。目前 RAG 系统的**最佳实践**，通常与方案 2 组合使用。详细原理见下方「Reranking 原理」章节。

**4. Metadata Filter + 分类器**

先用轻量分类器判断 query 领域，再在对应领域内检索。适合文档量大、领域多的场景。

**行业现状**：主流 RAG 系统（RAGFlow、LangChain 生产部署）基本采用 **方案 2 + 方案 3 组合**：BM25 + 向量混合召回 → Cross-Encoder Reranker 精排。

### Reranking 原理（重排序）

**为什么向量搜索不够准**：向量搜索（Bi-Encoder / 双塔模型）分别把 query 和文档编码成向量再算距离，**两者从未互相"看"过**。信息在编码阶段就已丢失。

**Reranker（Cross-Encoder / 交叉编码器）**：把 query 和文档拼在一起输入模型，让两者充分交互：

```
向量搜索(Bi-Encoder):
  query → [Encoder] → 向量A  对比  文档 → [Encoder] → 向量B  → 算距离

Reranker(Cross-Encoder):
  "[CLS] query [SEP] 文档 [SEP]" → [Encoder] → 直接输出相关度分数
```

**工作原理**：以 "怎么保证API安全认证" 和 "JWT是一种无状态的身份验证方案" 为例，BERT 类模型的自注意力机制（Self-Attention）让 query 中每个字看到文档中每个字：
- "安全" 关注到 "身份验证" → 强相关信号
- "认证" 关注到 "JWT" → 强相关信号
- 最后池化层输出相关度分数（如 0.92）

**Tradeoff**：
- 向量搜索：百万文档毫秒级（一次编码 + HNSW 查表）
- Reranker：每个候选文档跑一次推理，20 个候选 = 20 次推理，成本高
- **工程做法**：向量搜索粗筛 top 20 → Reranker 精排 top 3

### 常见误解澄清

- **混合检索 ≠ 让 LLM 合并结果**：BM25 和向量检索的结果通过 RRF 等算法合并，是纯数学操作，LLM 只在最后一步拿最终结果生成回答
- **BM25 不需要关键词提取**：直接把原始 query 传进去，BM25 自动分词并按 IDF 加权
- **Query Rewrite ≠ 混合检索的语义搜索部分**：前者用 LLM 改写 query 再检索，后者用原始 query 同时走两条路

### Metadata Filter 补充说明

**标签不是 Vector DB 自动生成的**——是入库前由你准备好的。Vector DB 只提供 `metadata` 存储位置，具体放什么标签、怎么分类，完全由你决定。

```python
# 入库时：自己准备标签
documents = [
    {"id": "doc_14", "content": "JWT 是一种无状态的身份验证方案", "category": "security", "type": "auth"}
]
# 存入时 metadata 和 embeddings 完全独立
collection.add(documents=texts, embeddings=embeddings, ids=ids, metadatas=metadatas)

# 检索时：用 filter 缩小范围
collection.query(query_embeddings=[vec], where={"category": "security"}, n_results=3)
```

**入库打标方式**：
- 手动标注：文档量少时直接写
- LLM 批量打标：入库前让模型判断类别
- 规则提取：文件名/路径包含关键词自动标

**Metadata Filter 的优化不在算法层面**（它就是个 `WHERE` 条件），而在**标签体系设计**和**分类器准确率**——标得准，过滤就准；标得乱，反而会把相关内容滤掉。

### 检索优化方案对比总结

| 方案 | 是否需要 LLM | 计算成本 | 适用场景 |
|------|-------------|---------|---------|
| Query Rewrite | 是 | 高（多一次 LLM 调用） | 模糊问题拆具体子问题 |
| Hybrid Search (BM25 + 向量) | 否 | 低（纯算法） | 互补召回盲区 |
| Reranking | 否（专用模型） | 中（多次推理） | 精排粗筛结果 |
| Metadata Filter | 否（入库时可能用） | 极低 | 文档量大、领域多 |

### PostgreSQL 作为向量数据库（pgvector）

PostgreSQL 通过 **pgvector** 扩展支持向量存储和搜索，越来越多项目选择用它替代专门的向量数据库。

**使用方式**：

```sql
CREATE EXTENSION vector;
CREATE TABLE doc_embeddings (
    id text PRIMARY KEY,
    content text,
    embedding vector(1024)  -- 维度需匹配 embedding 模型
);
CREATE INDEX ON doc_embeddings USING hnsw (embedding vector_cosine_ops);

-- 语义搜索（<=> 余弦距离, <#> 内积, <-> 欧氏距离）
SELECT id, content FROM doc_embeddings
ORDER BY embedding <=> '[0.1, 0.2, ...]'
LIMIT 5;
```

**为什么越来越多项目选 pgvector**：

1. **少一个基础设施**：业务数据 + 向量数据在一个库里，可 JOIN，零额外运维
2. **混合查询能力**：SQL 的 `WHERE`、子查询、JSONB、权限检查等直接用，远超向量 DB 的简单 key-value filter
3. **事务一致性**：文档增删改和向量更新在同一事务中
4. **成熟生态**：备份、复制、连接池等 20 年 Postgres 工具链复用

**Tradeoff**：

| 维度 | pgvector | 专用向量 DB（Pinecone/Qdrant）|
|------|---------|-----------|
| 部署 | 复用现有 Postgres | 新增一套服务 |
| 性能 | 百万~千万级够用 | 十亿级更强 |
| 功能 | HNSW/IVFFlat | 更丰富（动态向量、稀疏向量、多模态）|
| 适用场景 | 已有 Postgres 的项目 | 纯向量场景、超大规模 |

**结论**：已有 PostgreSQL 的项目，向量量级 < 1000 万时，pgvector 是最务实选择——门槛最低、集成成本最小。

### MySQL vs PostgreSQL 作为向量数据库

MySQL 也有向量搜索支持，但生态远不如 pgvector：

**MySQL 向量搜索历程**：
- 2024 年 MySQL 8.0.31 实验性支持向量距离函数
- 2024 年 MySQL 9.0.0 支持 `VECTOR()` 类型和 HNSW 索引
- 功能仍在快速迭代中，不如 pgvector 成熟

**为什么大家选 pgvector 不选 MySQL 向量**：

1. **时间差**：pgvector 2021 年发布，MySQL 向量 2024 年才可用，LangChain/LlamaIndex 等框架已默认支持 pgvector
2. **索引成熟度**：pgvector 的 HNSW 经过 3 年生产验证，MySQL 刚起步
3. **PostgreSQL 扩展机制**：pgvector 利用 Postgres 的 Extension 机制，支持自定义操作符（`<=>`、`<#>`）；MySQL 不支持扩展自定义操作符
4. **复杂查询能力**：PostgreSQL 的 JSONB、CTE、窗口函数在混合查询场景下比 MySQL 更强
5. **设计哲学**：PostgreSQL 是"可扩展的关系型数据库"，鼓励第三方扩展；MySQL 是"快、简单、够用"，扩展生态不如 Postgres

**结论**：pgvector 先发优势 + PostgreSQL 本身更适合"复杂查询 + 扩展"的基因，使其成为 RAG 领域的默认选择。

## Step 3：文档切分（Chunking）

### 为什么需要 Chunking

embedding 模型对一段文本只输出**一个向量**。如果把整篇文章（如 2000 字）塞进去，这个向量捕捉的是"全文的整体语义"，丢失了细节。比如问"计费多少钱"——整篇的 embedding 和查询的匹配度可能很低，因为被其他无关内容稀释了。

Chunking 的本质：**把文档拆成更细粒度的语义单元**，让每个 chunk 的 embedding 能精准对应一个具体问题。

### 三种切分策略的本质差异

| 策略 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| 固定字符数 | 按长度等分切 | 简单通用 | 可能把一句话/一个概念切成两半 |
| 按段落切分 | 按双换行(`\n\n`)分割 | 保持语义完整性，一个段落通常只讲一件事 | 段落长短不一，有的太短、有的太长 |
| 按标题切分 | 按 Markdown 标题(`##`)分割 | 语义最完整 | chunk 大小差异大，长的 chunk 又回到"信息太杂"问题 |

**实验数据**（1586 字符技术文档）：
- 固定字符数(500/50): 4 个 chunk，大小 236-500
- 按段落切分: 28 个 chunk，大小 10-146
- 按标题切分: 6 个 chunk，大小 14-618

### 检索质量对比实验

| 查询 | 固定字符数 | 按段落 | 按标题 | 最优策略 |
|------|-----------|--------|--------|---------|
| "百炼平台有哪些模型" | 0.8221 | **0.9077** | 0.8340 | 按段落 |
| "qwen-plus 计费标准" | 0.7913 | **0.8361** | 0.7278 | 按段落 |
| "如何处理 API 速率限制错误" | **0.6725** (100/0) | 0.7832 | 0.6458 | 按段落 |

**按段落切分在 3 个查询中均表现最佳**，因为它保持了语义完整性——每个段落只讲一件事，embedding 更精准。

### Chunk 大小与检索精度的反向关系

小 chunk 有时比大 chunk 更准，但前提是**语义完整性**。固定字符数 100/0 的切分虽然小，但经常把句子切成两半（如"## 2. 模..."），丢失语义反而不如 300/30。但当 chunk 足够小且语义集中时（如"速率限制"相关段落），小 chunk 的 embedding 不会被无关信息稀释，匹配度更高。

**核心规律**：不是 chunk 越小越准，而是**语义越集中的 chunk 越准**。

### Overlap 的作用

Overlap（重叠部分）防止关键信息被切分点割裂。

实验验证：构造一段文本，"qwen-plus 模型用于文本生成"和"qwen-plus 的计费标准"之间被填充了 50 个无意义字符，测试 query "qwen-plus 计费多少钱"。

- **无 overlap**：chunk[2] 相似度=0.7445（只有后半句，缺少上下文）
- **有 overlap(20)**：chunk[2] 相似度=0.8072（overlap 带入了前一个 chunk 的尾部"qwen-plus 模型"，语义更完整）

**为什么 overlap 能提升相似度**：overlap 让 chunk 包含了相邻 chunk 的部分内容，使得原本被切分点割裂的语义在 overlap 区域得到了补充。

**Tradeoff**：overlap 增加存储和计算成本——10% overlap 意味着存储量增加约 10%，但检索精度提升明显。

### 生产环境最佳实践

1. **结构化文档（Markdown/HTML）优先按标题/段落切分**
2. **非结构化文本**：使用 `RecursiveCharacterTextSplitter`（先按标题切，某节太长再递归按段落/字符切）
3. **chunk_size**：300-800 字符（中文 200-500 字）
4. **overlap**：10-15% 的 chunk_size

### 语义 Chunking（Semantic Chunking）

与普通 chunking（固定字符数、按段落）的**结构驱动**不同，语义 chunking 是**内容驱动**——根据文本语义变化点切分，而不是根据格式或长度。

**实现方式**：
1. 按句子切分成最小单元
2. 计算相邻句子的 embedding 相似度
3. 相似度突然下降的位置 = 语义转折点 = 切分点

```
句子1: "百炼平台支持多种模型"        ← 相似度高 → 合并
句子2: "包括 qwen-plus、qwen-max"   ← 相似度高 → 合并
句子3: "这些模型有不同价格"          ← 相似度下降 → 切分（话题变了）
句子4: "qwen-plus 是 0.004 元/token" ← 相似度高 → 合并
```

**另一种方式**：用 LLM 判断每句话主题，相同主题归为一个 chunk。

**Tradeoff**：

| | 固定字符数 | 按段落/标题 | 语义 chunking |
|--|-----------|------------|--------------|
| 切分依据 | 字数 | 格式 | 内容语义 |
| 语义完整性 | 差 | 中 | 好 |
| 计算成本 | 无 | 无 | 高（需计算大量 embedding 相似度）|

**生产现状**：语义 chunking 效果最好但成本高，大多数项目用"按标题/段落切分 + 超长递归"就够了。语义 chunking 更多用于文档格式不统一、结构混乱的场景。

### 语义 Chunking 的成本说明

语义 chunking 需要计算 N-1 次相邻句子 embedding 相似度，每次调用 embedding API **都会消耗 token**。

以 5000 字文档为例：
- 按句子切成 200 句 → 199 次 embedding 调用
- 每次平均 25 字 ≈ 约 10 token × 199 = ~1990 token
- 1000 篇文档就是 ~200 万 token（仅预处理阶段）

**成本高 = token 费用 + API 调用延迟**，两者都是实际成本。

### Embedding API 的计费与调用位置

**Embedding API 也是远程 API 调用**，不是本地计算：
```python
# 模型运行在阿里云服务器上，不是本地
response = client.embeddings.create(
    model="text-embedding-v3",
    input="这是一段文本",
)
```
- Embedding 模型（text-embedding-v3）和生成模型（qwen-plus）是两个不同的模型
- 两个都跑在远程服务器，都按 token 计费
- 类比：embedding API = 文字→向量（按输入 token 计费）；chat API = 生成回答（按输入+输出 token 计费）

**Dashscope 免费额度**：text-embedding-v3 有免费 token 额度（约 100 万/月，以官网为准），学习场景够用。

**RAG 两阶段调用**：
1. **索引阶段（一次性）**：文档 → 切分 → 生成 embedding → 存 Vector DB，每个 chunk 调一次 embedding API
2. **查询阶段（每次查询）**：用户问题（单句）→ 直接 embedding（不需要切分）→ 搜 Vector DB，调一次 embedding API

**查询不需要切分的原因**：用户问题通常很短（一句话），本身语义完整，切分反而会丢失语义。只有长文档才需要 chunking。

### 复杂文档（PDF/代码/配置文件）的 Chunking 策略

生产环境中处理多份复杂技术文档的正确做法：

**1. 先识别内容类型，再决定切分方式，不一刀切**：
```
PDF 解析 → 识别内容类型
  ├── 普通段落 → 按段落/标题切分
  ├── 代码块 → 保持完整，单独一个 chunk
  ├── 表格 → 保持完整，单独一个 chunk
  ├── 图片 → 提取描述/OCR，走多模态处理（如 qwen-vl）
  └── 标题 → 提取为 metadata tag
```

**2. Tag 打在章节/段落级别，不是文档级别**：一份 PDF 可能跨多个主题，整份文档打一个 tag 太宽泛。metadata filter 需要精准的章节级标签。

**3. 代码块保持完整**：固定字符数切分会把函数切成两半，丢失语义。应按函数/类定义切分。

**4. 长代码（>200 行）**：按函数/类边界切分，每个函数一个 chunk，检索到的是完整的代码单元。

**5. 短脚本/配置文件（几十行以内）**：如 shell 脚本、短 SQL、小配置文件 → 整个当 chunk，不切。按执行顺序写的完整逻辑，切开失去意义，且成本低。

**6. 长配置文件（如几百行 Kubernetes YAML）**：按 YAML 的 `---` 文档分隔符切分，每个 section 一个 chunk。

## Step 4：最简 RAG Pipeline

### RAG 三阶段流程

```
1. 索引阶段（一次性）: 文档 → 切分 → embedding → 存 Vector DB
2. 检索阶段（每次查询）: 用户问题 → embedding → 搜索相似 chunk
3. 生成阶段: 检索结果 + prompt → LLM 生成回答
```

**关键理解**：**LLM 只在最后一步出现**，前两步都不需要 LLM 生成。这是 RAG 和"直接问 LLM"的本质区别。

### RAG Prompt 设计

```
请基于以下参考资料回答用户问题。
参考资料：{retrieved_chunks}
要求：
1. 回答必须基于参考资料，不要编造信息
2. 如果参考资料中没有相关信息，请明确说"我没有找到相关信息"
3. 引用参考资料时用 [1]、[2] 这样的标注
用户问题：{query}
```

这个 prompt 做了三件事：
- **限定知识来源**：把检索到的 chunk 给 LLM，减少幻觉（Hallucination）
- **要求标注引用**：回答中用 [1]、[2] 标注，让回答可追溯（Traceable）
- **要求说"不知道"**：防止知识库里没有时瞎编

### 检索质量 = RAG 的上限

**核心原则**：检索错了，后面全错。LLM 拿到不相关的参考资料，生成的回答也会受影响。

实验观察（8 篇文档知识库）：
- "百炼平台有哪些模型？"：第一名距离 0.3001（精准命中），第二名距离 1.0718（完全不相关，只因提到"百炼"和"格式"）
- "qwen-plus 多少钱？"：第一名距离 0.5524（精准），第二名距离 0.7172（相关但非直接答案）
- 同一个 query，不同文档的距离差异大，取决于文档语义和 query 的匹配程度

### 正常问答 vs 知识库里没有的问题

实验验证了 RAG 系统对两类问题的处理：

| 问题类型 | 示例 | 检索距离 | 回答 |
|---------|------|---------|------|
| 知识库有答案 | "qwen-plus 多少钱？" | 0.55, 0.72 | 基于参考资料准确回答 |
| 知识库有答案 | "API 调用频率限制怎么解决？" | 0.70, 0.76 | 综合两篇参考回答 |
| 知识库没有 | "Python 的装饰器怎么写？" | 1.01, 1.01 | "根据现有资料，我没有找到相关信息" |
| 知识库没有 | "Docker 怎么部署？" | 0.94, 1.11 | "根据现有资料，我没有找到相关信息" |

**没有答案时模型如何说"不知道"**：Prompt 中明确要求 + 模型本身有遵循指令的能力。但如果 prompt 中没有这个约束，模型可能会基于自己的训练数据回答（产生幻觉）。

### 当前 Pipeline 的不足（后续 Step 改进方向）

- 没有 chunk_overlap（检索可能漏信息）
- 固定 top_k=3，不根据 query 调整
- 没有评估机制，不知道回答质量
- 只用了语义搜索，没有混合检索

## Step 5：RAG 质量评估

### LLM-as-Judge 评估框架

用 LLM 当裁判，从三个维度对 RAG 系统评分：

| 维度 | 评估什么 | 理想情况 |
|------|---------|---------|
| Context Precision（检索准确率） | 检索到的 chunk 是否包含答案相关信息 | 检索到的都是相关文档 |
| Faithfulness（忠实度） | 回答是否基于检索内容，没有编造 | 回答完全来自参考资料 |
| Answer Relevance（回答相关性） | 回答是否真正回答了用户问题 | 完整准确回答了问题 |

### 评估实验结果（16 个测试用例，8 篇文档知识库）

| 维度 | 平均分 | 分析 |
|------|-------|------|
| Context Precision | **3.88/5.00** | 检索有时不准，尤其是 query 与文档标题不匹配时 |
| Faithfulness | **4.94/5.00** | 几乎满分，说明 prompt 约束有效，LLM 没有瞎编 |
| Answer Relevance | **3.44/5.00** | 最低维度，因为知识库没有答案时回答是"不知道" |
| 综合平均 | **4.08/5.00** | |

**为什么 Faithfulness 最高**：Prompt 中明确要求"基于参考资料回答"，LLM 遵守得很好。8/16 的 Faithfulness 满分。

**为什么 Answer Relevance 最低**：知识库没有答案的问题（如"Python 装饰器怎么写"），回答是"没有找到相关信息"，忠实度 5/5（没瞎编），但相关性 1/5（没回答问题）。这是**RAG 系统的固有矛盾**——忠实和正确之间的权衡。

### 好 Case vs 差 Case 分析

**表现最好的 Case（综合 ≥ 4.5）**：8 个，如"qwen-plus 多少钱"、"百炼平台有哪些模型"、"429 错误码什么意思"
- 共同规律：query 措辞和文档标题/内容高度匹配
- 检索精准，回答准确

**表现最差的 Case（综合 ≤ 3.0）**：4 个

| 问题 | Context | Faithful | Relevant | 根因 |
|------|---------|---------|---------|------|
| "Prompt Engineering 最佳实践" | 1/5 | 5/5 | 1/5 | 检索完全错误，搜到无关文档 |
| "Python 的装饰器怎么写" | 1/5 | 5/5 | 1/5 | 知识库里没有 |
| "GPT-4 和 qwen-max 哪个更好" | 3/5 | 5/5 | 1/5 | 知识库里没有 GPT-4 信息 |
| "Docker 怎么部署" | 1/5 | 5/5 | 1/5 | 知识库里没有 |

### 关键发现：检索错了，后面全错

"Prompt Engineering 最佳实践"是最佳例证：
- 检索到的是"模型选型"和"API 速率限制"（完全不相关）→ Context Precision 1/5
- LLM 诚实地说"没有找到相关信息" → Faithfulness 5/5（没编造）
- 但用户问了问题，回答"不知道" → Answer Relevance 1/5

**链条**：检索失败 → 没有参考 → 回答"不知道" → 用户不满意

**改进方向**：如果检索正确（搜到了"Prompt Engineering 最佳实践"文档），Context Precision 会从 1 变成 5，Answer Relevance 也会跟着提升。这再次证明**检索质量是 RAG 的上限**。

### 抽象词检索问题

query 中的抽象词（"最佳"、"怎么"、"如何"）会把 embedding 向量推向错误方向。例如"Prompt Engineering 最佳实践"：
- doc_07 标题含"最佳实践"，但**内容里没有这四个字**
- 内容讲的是"few-shot 示例"、"function calling"等具体技巧
- embedding 模型把"最佳实践"理解成了"建议/推荐"
- 其他文档中的"推荐默认选择"、"申请提升配额"反而更匹配

**三种解法**：

| 方案 | 原理 | 适用场景 |
|------|------|---------|
| Query Rewrite | 把"最佳实践"改写为"技巧"，去掉抽象词 | 通用，成本一次 LLM 调用 |
| 低置信度提示 | 检索结果相似度都 < 阈值时，提示用户换问法 | 用户体验好，需设阈值 |
| 优化文档内容 | 在文档内容中也加入标题关键词，让 embedding 更好绑定 | 适用于可控知识库 |

### LLM-as-Judge 评分宽松问题

实验中发现 LLM 评分普遍偏高（Faithfulness 4.94/5.00），这是**已知问题**：

**原因**：
- LLM 有"讨好"倾向（sycophancy），倾向于给高分
- 评分 prompt 里没有"标准答案"对比，只让基于参考资料判断
- 较弱模型（qwen-plus）当裁判时偏温和

**生产中的解法**：
1. **用更强的模型当裁判**：gpt-4o、Claude Opus 评分更严格
2. **提供明确的评分 rubric**：不只"1-5 分"，而是每一分有具体标准（如"5分=包含所有3个关键信息，4分=包含2个，3分=包含1个"）
3. **混合评分**：LLM-as-Judge + 关键词匹配（检查回答是否包含 expected_key_points 中的关键词）

## Step 6：RAG 进阶技巧

### Query Rewriting（查询重写）

让 LLM 把模糊 query 改写为更具体的子问题，再去检索：

```python
# 原始: "Prompt Engineering 有什么最佳实践？"
# 重写: ["Prompt Engineering 技巧", "few-shot 示例方法", "function calling 用法"]
```

实现方式：prompt 要求 LLM 去掉抽象词（"最佳"、"怎么"），使用具体技术术语。多个改写后的 query 分别搜索，合并去重后取 top-k。

**Tradeoff**：多一次 LLM 调用，增加延迟和成本。

### Re-ranking（重排序）

先放宽召回（expand_k=5），再用 LLM 对候选文档做相关度排序，取 top_k：

```python
# 第一步：语义搜索 5 个候选
candidates = vector_search(query, n_results=5)
# 第二步：LLM 评估每个候选与 query 的相关度，排序
ranked = rerank(query, candidates)
# 第三步：取前 3 个
return ranked[:3]
```

**Tradeoff**：多一次 LLM 推理，但只跑了 1 次（评 5 个文档），不是 5 次。

### 优化文档内容（修复检索失败）

Step 5 中"Prompt Engineering 最佳实践"检索失败，原因是 doc_07 的标题有"最佳实践"但内容里没有这四个字。

修复方法：在内容开头加上"Prompt Engineering 最佳实践包括以下技巧"，让标题和内容的关键词对齐。

修复后效果：Step 5 的 1/5 → Step 6 的 5/5。**这是最简单的优化，但只适用于你能控制知识库的场景。**

### 三种策略对比实验

| 策略 | 平均分 | 好结果(≥4) | 说明 |
|------|-------|-----------|------|
| 基线（直接语义搜索） | 4.77/5 | 13/13 | 小知识库基线已很强 |
| Query Rewriting | 4.77/5 | 13/13 | 持平——小知识库 query 已够准 |
| Re-ranking | **4.85/5** | 13/13 | 略微提升——排序更精准 |

**结论**：
- 在小知识库（8 篇）中，基线已很强，进阶技巧提升有限
- Re-ranking 是性价比最高的优化——只需多一次 LLM 调用，排序效果改善
- Query Rewriting 在大规模知识库中效果更明显
- **优化文档内容是最简单有效的优化**，但只适用于可控知识库

### RAGAS 评估框架

RAGAS（RAG Assessment）是专门的 RAG 评估 Python 库：

```bash
pip install ragas
```

**核心指标**：

| 指标 | 评估什么 |
|------|---------|
| Faithfulness | 回答是否基于检索内容（分解为声明逐个验证）|
| Answer Relevance | 回答是否切题（从回答生成假设问题匹配）|
| Context Precision | 检索到的 chunk 是否相关（按排名加权）|
| **Context Recall** | 检索是否召回了所有必要信息（对比参考答案）|

**和手写 LLM-as-Judge 的区别**：
- RAGAS 的 Faithfulness 更严格：把回答分解为独立声明，逐个检查是否可从上下文推导
- RAGAS 有 **Context Recall**：需要参考答案（ground truth），评估"检索覆盖了多少关键信息"
- 内置评估 pipeline + 统计报告，但学习成本更高

**基本用法**：
```python
from ragas import evaluate
from ragas.metrics import faithfulness, answer_relevance, context_precision

result = evaluate(dataset, metrics=[faithfulness, answer_relevance, context_precision])
```

**RAGAS 分数反映的是哪个环节的问题**：

| 指标组合 | 说明 |
|---------|------|
| Context Precision/Recall 低 + Faithfulness 高 | 检索错了，但 LLM 很老实 → 优化检索策略 |
| Context Precision 高 + Faithfulness 低 | 检索对了，但 LLM 在瞎编 → 优化 prompt 约束 |
| Context Recall 低 + Answer Relevance 低 | 漏了关键文档，回答不完整 → 增加 top_k 或换检索方式 |
| 四个都高 | 整体都好 |

### Chunking vs 检索优化：哪个收益更大？

**结论：Chunking 是地基，检索优化是锦上添花。**

| 优化环节 | 提升幅度 | 说明 |
|---------|---------|------|
| 差 Chunk → 好 Chunk | **+30~40分** | 质的飞跃，Garbage In, Garbage Out |
| 好 Chunk → 加混合检索 | +5~10分 | 锦上添花 |
| 混合检索 → 加 Rerank | +3~5分 | 边际改善 |

**类比**：好 Chunk = 图书馆的书分类正确、内容完整；好检索 = 高级索引系统。索引再先进，也弥补不了书被撕了半页的问题。

**正确优化顺序**：Chunking → 探针测试 → 检索策略 → Reranking。每步做完再考虑下一步。
