## 潜在成长方向：AI Agent 测试与评估

> 2026-05-09 补充。用户对方向本身感兴趣，作为备选方向记录。

### 方向概述

AI Agent 的测试与评估是 AI 工程化过程中的新兴领域，核心工作是**系统化、自动化地验证 Agent 的行为质量和可靠性**。由于 LLM 输出具有概率性和非确定性，传统测试的"精确断言"范式不再适用，需要全新的评估方法论和工程体系。

### 核心工作内容

| 工作领域 | 具体内容 |
|----------|----------|
| 输出质量评估 | 用裁判模型（LLM-as-judge）或评分规则判断准确性、完整性、安全性、格式合规性 |
| 评估数据集构建 | 收集真实对话数据，构建数百至数千条的基准测试集，用于回归验证 |
| 自动化评估流水线 | 将评估集集成到 CI/CD，Agent 迭代后自动跑分、出报告、低于阈值阻断发布 |
| 工具调用验证 | 验证 Agent 是否正确选择工具、参数是否准确、调用顺序是否合理、失败后降级行为是否正常 |
| 长程行为监控 | 多步任务的中间状态追踪、错误定位、思考过程的观测与调试 |

### 与传统测试的差异

| 维度 | 传统测试 | Agent 测试 |
|------|----------|------------|
| 断言方式 | 精确匹配（assertEquals） | 语义评分 / 模型裁判 |
| 回归标准 | 确定性 pass/fail | 统计意义上的趋势分析 |
| 新挑战 | — | 幻觉检测、安全性评估、prompt 注入测试、非确定性重试 |
| CI/CD | 跑测试脚本 | 跑评估集 + 评分流水线 + 趋势看板 |

### 为什么适合用户

- **14年测试开发经验**：自动化测试框架设计、CI/CD 集成、质量评估体系可直接迁移
- **差异化竞争**：纯做 AI 的人不懂系统工程，做测试的人不懂 LLM 特性，两者兼备的人极少
- **市场需求**：企业级 Agent 落地后，评估和可靠性是刚需，但目前市场上缺乏专业人才

### 学习路径上的侧重点调整

在原有 5 阶段学习路径基础上：

- 第三阶段 RAG 完成后 → 增加**"给这个 RAG 构建评估集并跑分"**的实践
- 第四阶段 Agent 完成后 → 增加**"用 LLM-as-judge 自动化评分"**的实践
- 整体多关注：评估指标设计、评分自动化、数据集构建、趋势分析

### 需要补充的技能

- 评估框架工具：Ragas、DeepEval、LangSmith 等

## 建议学习路径（基于画像）

> 以下为基于评估的建议，非强制。

### 第一阶段：API 实操（1-2周）
目标：打破"从未调用过 API"的零经验状态
- 注册 OpenAI 或 Claude API，用 Python 写第一个 LLM 调用
- 学习 system/user/assistant 消息格式
- 学习 Function Calling / Tool Use 的 JSON schema 定义方式

### 第二阶段：Prompt Engineering（1周）
目标：从"想到什么写什么"到结构化 prompt
- few-shot prompting、CoT、structured output
- 每个技巧都写代码验证效果

### 第三阶段：RAG 入门（2-3周）
目标：实现一个最简 RAG 系统
- 理解 Embedding 原理
- 用 Chroma/Qdrant 等轻量 Vector DB
- 用 Python + Flask 做一个本地知识问答 Demo

### 第四阶段：Agent 框架（2-3周）
目标：用框架搭建多步骤 Agent
- LangChain 或 LlamaIndex 基础
- 实现一个能调用多个工具、做条件判断的 Agent
- 对比"手写 Agent loop" vs "框架 Agent"的差异

### 第五阶段：探索方向（持续）
- 企业内部工具/自动化 Agent
- 开发者工具/DevOps Agent
- 消费者产品 Agent
- **特别关注**：Agent 测试与评估（与 QA 背景强相关）
